Full review comments:

- [P1] Parse Claude result envelopes before returning no findings — D:\extension\src\adapters\claude-adapter.js:149-149
  The repo's own spike evidence shows `claude --output-format json` returning a wrapper object whose `result` field is plain text (`docs/spike-results-v3.json`). In that normal case `_tryParseJson()` reaches this `return []`, so `parseResult()` never falls back to `_parseTextOutput()`. Any Claude review that reports issues inside `result` will therefore be recorded as zero findings, effectively disabling the Claude reviewer.

- [P2] Decode agent output as a UTF-8 stream, not per chunk — D:\extension\src\adapters\base-adapter.js:206-206
  A subprocess `data` event can end in the middle of a multibyte UTF-8 codepoint. Decoding each buffer with `toString('utf-8')` can insert `�` for perfectly valid Vietnamese/Japanese text, and the subsequent garble check then drops that whole chunk from `allOutput`. In the UTF-8 scenarios this repo explicitly tests, that can silently lose findings or make parsing nondeterministic; a streaming decoder (`StringDecoder`/`TextDecoder`) is needed here.

- [P2] Normalize finding paths before creating dedupe keys — D:\extension\src\adapters\codex-adapter.js:112-112
  This passes `raw.file`/`raw.path` straight into `createFinding()` even though `normalizeFindingPath()` was added for this exact case.
